# Treasury Multi-Agent Committee 改善計画書

## このシステムの価値は「出力の品質」で決まる

どれだけUIが綺麗でも、セキュリティが万全でも、審議結果がカスなら誰も使わない。改善の最優先は**審議結果の品質**。使い物になるレポートが出るようになってから、その他を磨く。

---

## 現状の問題：なぜ結果がカスになるのか

### 問題1: プロンプトが浅い
**現状:** 各ペルソナのシステムプロンプトは「こういう観点で評価して」程度の汎用的な指示。具体的な分析フレームワーク、数値基準、判断ロジックが欠けている。
**結果:** どのモデルを使っても「一般論の羅列」になり、企業固有の深い分析にならない。

### 問題2: ペルソナ間の議論が噛み合わない
**現状:** 反論ラウンドで「他の専門家の見解に反論してください」としか指示していない。前のステップの出力をテキストで丸ごと渡しているだけ。
**結果:** 表面的な反論か、単なる繰り返しになる。建設的な議論にならない。

### 問題3: ケース情報の活用が雑
**現状:** 企業の財務データ（現預金、時価総額、純有利子負債）を文字列でプロンプトに埋め込んでいるだけ。
**結果:** 「現預金$200Mの企業」と「現預金$50Mの企業」で分析の質が変わらない。財務比率の計算、同業比較、リスク許容度の数値的な検証がない。

### 問題4: 最終レポートの構造が貧弱
**現状:** エディターペルソナに「整形してください」と丸投げ。レポートの品質がモデルの気分次第。
**結果:** セクションの抜け漏れ、論理の飛躍、結論と根拠の不整合が起きる。

### 問題5: コンテキストが膨れて品質劣化
**現状:** 後半のステップほど、前の全出力をプロンプトに詰め込む。コンテキストが膨大になる。
**結果:** モデルが重要情報を見失い、後半ほど品質が下がる。特にChair統合と最終レポートで顕著。

## P0: 審議品質の根本改善

### 1. プロンプトエンジニアリングの本格化

**現状→改善:**

#### BTC専門家
- 現状: 「BTCのトレジャリー適合性を評価して」
- 改善: 具体的な分析フレームワークを指定
  - FASB ASU 2023-08 に基づく会計処理の具体的影響
  - 「現預金比率 X% の企業がBTCを Y% 保有した場合」の数値シミュレーション指示
  - MicroStrategy, Tesla, Block 等の先行事例との比較を要求
  - ボラティリティ定量化（30日/90日/365日 realized vol）の提示を要求

#### Risk Officer
- 現状: 「反対意見を出して」
- 改善: 具体的なストレステストシナリオを指定
  - 「BTCが70%下落した場合のB/Sインパクト」を計算させる
  - 流動性カバレッジ比率への影響を定量的に示させる
  - 監査法人・格付機関の反応を予測させる

#### Chair
- 現状: 「統合して最終提案を作って」
- 改善: 意思決定マトリクスの構築を指示
  - 各専門家の推奨を比較表にまとめさせる
  - 賛成/反対の論拠を重み付け評価させる
  - 「もし取締役会に1枚のスライドで説明するなら」の要約を要求

### 2. 構造化出力の強制

**現状:** LLMに自由形式で返答させ、JSONであることを祈っている。
**改善:**
- OpenRouter の JSON mode / structured output を活用
- 各ペルソナの出力をZodスキーマで定義し、パース失敗時はリトライ
- 必須フィールドが欠けていたら再生成を要求
- confidence の数値根拠（なぜ0.7なのか）を必ず説明させる

### 3. ケースデータの事前分析

**現状:** 生の数値をプロンプトに貼るだけ。
**改善:** プロンプトに渡す前にサーバー側で財務指標を計算:
- 現預金/時価総額比率
- 現預金/純有利子負債比率
- 候補資産を最大許容DDまで保有した場合の最大損失額
- 最小流動性確保に必要な現預金額

これらを計算済みの数値としてプロンプトに含めることで、LLMが「計算」する必要をなくし、「判断」に集中させる。

### 4. 反論ラウンドの再設計

**現状:** 「他の意見に反論して」→ 表面的な反論。
**改善:**
- 各専門家に「他の専門家の主張で最も弱い点を1つ特定し、データで反証せよ」と指示
- 反論対象を明示的に指定（BTC専門家にはGold専門家の意見を、Gold専門家にはBTC専門家の意見を）
- 「あなたの推奨を変更する条件は何か」を必ず述べさせる

### 5. コンテキスト圧縮

**現状:** 全ステップの生出力をそのまま次のプロンプトに渡す。
**改善:**
- 各ステップの出力から構造化データ（summary, key_arguments, recommendation, confidence）だけを抽出して次に渡す
- 生テキストではなく要約済みの箇条書きで渡す
- コンテキスト長がモデルの80%を超えたら自動で圧縮

## P1: 出力をそのまま使えるレベルにする

### 6. レポートテンプレートの精緻化

**現状:** エディターに「整形して」と丸投げ。
**改善:** レポートのMarkdownテンプレートをコード側で定義し、エディターは各セクションの「内容」だけを生成:

```markdown
# 投資委員会メモ
## 1. Executive Summary（3文以内）
## 2. 審議対象と前提条件（表形式）
## 3. マクロ環境認識（箇条書き5項目以内）
## 4. 資産別評価（資産ごとにサブセクション）
### 4.1 [資産名] - 推奨: [badge] / 確信度: [X%]
## 5. リスクアセスメント（Red Flags表）
## 6. 推奨アロケーション（表 + 根拠）
## 7. 見送り理由（番号付きリスト）
## 8. モニタリングKPI（表：指標/閾値/頻度）
## 9. 次回審議トリガー条件
```

### 7. 専門家出力のリッチ表示

**現状:** JSONをpre タグで表示。
**改善:** 構造化された出力をカード形式で表示:
- Summary をヘッダーに
- Key Arguments を箇条書きカードに
- Risks をアラートカードに
- Recommendation をバッジ（色分け: 積極採用=緑、条件付き=黄、見送り=オレンジ、反対=赤）
- Confidence をプログレスバーに

### 8. トークン消費・コスト表示

**現状:** トークン数を捨てている。
**改善:**
- 各ステップの input/output トークン数を保存
- 審議全体のコスト概算を表示（モデル単価 × トークン数）
- ペルソナ別のコスト内訳
- 「この審議にいくらかかったか」が一目でわかる

### 9. 実行時のタイムアウトとリトライ

**現状:** 無限ハング or 1回失敗で全体死亡。
**改善:**
- 各LLM呼び出しに60秒タイムアウト
- Rate Limit (429) / Server Error (5xx) は3回リトライ（指数バックオフ）
- 1ステップ失敗しても残りは継続（部分結果保持）
- 失敗ステップの個別再実行ボタン

## P2: 使い勝手の向上

### 10. 人間の介入ポイント

**現状:** 「審議開始」→ 完全自動 →「レポート出ました」
**改善:**
- 一次意見の後に「ここまでの意見を確認して、追加の指示があれば入力」のステップ
- Chair統合の前に「反論とリスクを確認して、重視すべき論点を指定」
- 最終レポートに人間がコメントを追記して保存

### 11. ケーステンプレートと複製

**現状:** 毎回ゼロから入力。
**改善:**
- プリセットテンプレート（上場テック / 中堅メーカー / スタートアップ / 金融機関）
- 既存ケースの「複製して編集」ボタン
- 前回ケースからの差分だけ変更して再審議

### 12. 比較ビュー

**現状:** 1回の審議結果しか見られない。
**改善:**
- 同じケースで条件を変えた2回の審議を左右に並べて比較
- 「BTC 3% vs 5% でリスク評価はどう変わるか」が一目でわかる
- 推奨アロケーションの差分ハイライト

### 13. 外部データ連携

**現状:** すべて手入力。
**改善:**
- BTC/Gold/為替の現在価格を自動取得してケース情報に注入
- マクロ指標（CPI, 実質金利, VIX, DXY）を自動取得
- 「最新データで再審議」ボタン

### 14. ダッシュボード強化

**現状:** ケース数だけ。
**改善:**
- 最新の審議結果サマリ
- 実行中の審議進捗
- 累計コスト・トークン消費
- 直近の審議へのクイックリンク

## P3: 基盤整備

### 15. セキュリティ
- Firestore ルールの実装（ユーザースコープ）
- APIキーのマスク・サーバーサイド限定
- 管理者ロールの導入

### 16. テスト
- 審議フローのE2Eテスト（ダミーLLM使用）
- プロンプトテンプレートの回帰テスト
- CI/CD パイプライン

### 17. デプロイ自動化
- GitHub Actions で main push → Cloud Run 自動デプロイ
- Staging/Production 環境分離

### 18. ストリーミング
- SSE でLLM応答をリアルタイム表示
- ポーリングからWebSocketへの移行

## 優先度まとめ

| 優先度 | テーマ | 項目 |
|--------|--------|------|
| **P0** | **審議品質** | プロンプト本格化、構造化出力、ケースデータ事前分析、反論再設計、コンテキスト圧縮 |
| **P1** | **出力品質** | レポートテンプレート、リッチ表示、コスト追跡、タイムアウト/リトライ |
| **P2** | **使い勝手** | 人間介入、テンプレート/複製、比較ビュー、外部データ、ダッシュボード |
| **P3** | **基盤** | セキュリティ、テスト、CI/CD、ストリーミング |

**まず P0 をやる。P0 が終わって「このレポート、意思決定に使えるな」と思えるようになってから P1 以降に進む。**
