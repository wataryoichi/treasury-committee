# Treasury Multi-Agent Committee 改善計画書

## 現状サマリ

Next.js 16 + Firebase + 複数AIプロバイダーで構築されたAI投資委員会シミュレーター。6つのAIペルソナ（議長・BTC専門家・Gold専門家・マクロ・リスクオフィサー・エディター）が構造化された審議フローで投資判断を支援する。

**動いているもの:**
- ケースCRUD + サンプルデータ
- 6ペルソナの設定・ON/OFF・プロンプト編集
- Chair主導の6段階審議フロー（論点定義→一次意見→反論→リスク評価→統合→レポート）
- マルチプロバイダーLLM（OpenRouter/Anthropic/OpenAI/LM Studio/ダミー）
- ペルソナ別モデルオーバーライド（OpenRouter時）
- リアルタイム進捗表示（3秒ポーリング）
- ステップ出力の展開表示
- Markdown エクスポート（レポートのみ / 完全版）
- Firebase認証（Email/Password）
- Firebase Emulator による本番分離のローカル開発
- Cloud Run デプロイ + Firebase Hosting プロキシ

**公開URL:** https://treasury-committee.web.app
**GitHub:** https://github.com/wataryoichi/treasury-committee

## 1. セキュリティ（最優先）

### 1-1. Firestore セキュリティルールが全開放

**現状:** `allow read, write: if true;` — 誰でも全データを読み書きできる。
**影響:** 他人のケース閲覧・削除、APIキー窃取、データ改ざんが可能。
**対策:**
- ケースに `createdByUid` を付与し、作成者のみ読み書き可能に
- ペルソナ設定は認証済みユーザーのみ読み取り、管理者のみ書き込み
- モデル設定（APIキー含む）は管理者のみアクセス可能に
- Firestore Rules をきちんと実装してデプロイ

### 1-2. APIキーが誰でも読める

**現状:** `/api/settings` がAPIキーをそのまま返す。認証チェックなし。
**影響:** ログインした誰でもOpenRouterのAPIキーを取得できる。
**対策:**
- Settings APIに管理者権限チェックを追加
- GETレスポンスでAPIキーをマスクして返す（`sk-or-...***`）
- APIキーはサーバー側のみで使用し、クライアントに返さない

### 1-3. プロンプトインジェクション

**現状:** ペルソナのシステムプロンプトをUIから自由に編集できる。
**影響:** 悪意あるプロンプトで意図しない動作を引き起こす可能性。
**対策:**
- プロンプト編集を管理者のみに制限
- テンプレート方式でユーザーはパラメータだけ変更可能に

## 2. 審議エンジンの品質向上

### 2-1. エラー時の部分継続

**現状:** 1ステップでも失敗すると審議全体が `failed` になる。
**改善:** 失敗ステップをスキップして継続。部分結果を保持。個別ステップの再実行機能。

### 2-2. LLMレスポンスの構造化検証

**現状:** LLMの出力をそのまま保存。JSONとして不正でも気づかない。
**改善:** Zod でレスポンスをパースし、必須フィールド（summary, key_arguments, risks, recommendation, confidence）が揃っているか検証。不正な場合はリトライ or エラー表示。

### 2-3. コンテキストウィンドウの管理

**現状:** 各ステップで前の全出力をプロンプトに詰め込む。モデルのコンテキスト上限を考慮していない。
**改善:** 選択モデルの `contextK` と現在のプロンプト長を比較。超過する場合は要約して渡す or 警告を表示。

### 2-4. タイムアウトとリトライ

**現状:** LLM APIへのfetchにタイムアウトなし。無限ハング の可能性。
**改善:**
- 各API呼び出しに60秒タイムアウト
- 一時的エラー（429 Rate Limit, 503）は指数バックオフで3回リトライ
- タイムアウト時はステップを `failed` にして次へ進む

### 2-5. 同時実行制御

**現状:** ユーザーが何回でも「審議開始」を押せる。複数の審議が同時に走る。
**改善:** 同じケースで実行中の審議がある場合はブロック。ユーザーあたりの同時実行数を制限。

## 3. ユーザー体験の向上

### 3-1. ダッシュボードの強化

**現状:** ケース数のみ表示。「最近の審議」は常に `-`。
**改善:**
- 最新の審議結果サマリを表示
- 実行中の審議数をリアルタイム表示
- 直近の審議へのクイックリンク
- 累計トークン消費・コスト概算

### 3-2. ステップ出力のリッチ表示

**現状:** JSON をそのまま `<pre>` タグで表示。読みにくい。
**改善:**
- specialist_view の構造化データをカード形式で表示（summary, key_arguments, risks をそれぞれ整理）
- recommendation をバッジで色分け（積極採用=緑、条件付き=黄、見送り=赤）
- confidence をプログレスバーで可視化

### 3-3. 審議のリアルタイムストリーミング

**現状:** 3秒ポーリング。LLMが応答中は何も見えない。
**改善:** Server-Sent Events (SSE) で各ステップの出力をリアルタイムにストリーム表示。LLMの応答をトークン単位で表示。

### 3-4. 比較ビュー

**現状:** ケースごとに1回の審議結果しか見られない。条件を変えた比較ができない。
**改善:**
- 同じケースの複数回の審議結果を並べて比較
- 異なるケース間の推奨アロケーションを比較
- 条件変更前後のdiff表示

### 3-5. ケーステンプレート

**現状:** 毎回ゼロから入力。サンプルデータはAPI経由で手動投入。
**改善:**
- 「上場テック企業」「中堅メーカー」「スタートアップ」等のプリセットテンプレート
- 既存ケースの複製ボタン
- 入力のオートコンプリート

### 3-6. トークン消費・コスト追跡

**現状:** LLMレスポンスにトークン数が含まれるが、捨てている。
**改善:**
- 各ステップのトークン使用量を RunStep に保存
- 審議全体のコスト概算を表示（モデル単価 × トークン数）
- ダッシュボードに月間コスト推移グラフ

## 4. データ・運用

### 4-1. カスケード削除

**現状:** ケースを削除しても関連する Run と RunStep が孤児として残る。
**改善:** ケース削除時に関連 Run → RunStep を一括削除（Firestore batch delete）。

### 4-2. 監査ログ

**現状:** 誰がいつ何を変更したか記録がない。
**改善:**
- ケースの作成・更新・削除を audit コレクションに記録
- ペルソナ設定変更のログ
- モデル設定変更のログ

### 4-3. Firestore インデックス

**現状:** 複合クエリ（`where` + `orderBy`）用のインデックスが未定義。本番で一部クエリが失敗する可能性。
**改善:** 必要なインデックスを `firestore.indexes.json` に定義してデプロイ。

### 4-4. バックアップ

**現状:** Firestore のバックアップなし。
**改善:** Cloud Scheduler で日次バックアップを GCS に自動エクスポート。

## 5. 審議フローの拡張

### 5-1. カスタムワークフロー

**現状:** 6段階固定（論点定義→一次意見→反論→リスク→統合→レポート）。
**改善:** ワークフローエディターで段階の追加・削除・順序変更が可能に。例えば：
- 「第2回反論」を追加
- 「外部データ収集」ステップを挿入
- 「法務レビュー」ペルソナを追加

### 5-2. 人間の介入ポイント

**現状:** 審議は完全自動。人間の修正・フィードバックなし。
**改善:**
- 各ステップ完了後に「承認/修正/再実行」の選択肢
- 人間がコメントを追加してから次のステップへ
- Chair の最終提案に人間がフィードバックして再統合

### 5-3. 外部データ連携

**現状:** すべて手入力。市場データなし。
**改善:**
- BTC/Gold/為替の現在価格を自動取得
- マクロ指標（CPI, 実質金利, VIX）の自動注入
- 会社の財務データをAPI or CSV取り込み

### 5-4. 多言語対応

**現状:** プロンプト・UIともに日本語固定。
**改善:** プロンプトテンプレートに言語パラメータ追加。UIはi18n対応。英語/日本語切替。

## 6. 技術的改善

### 6-1. テスト追加

**現状:** テスト0件。
**改善:**
- LLMクライアントのユニットテスト（モック使用）
- API Route のインテグレーションテスト
- 審議フローのE2Eテスト（ダミーLLM使用）
- CI/CD パイプラインでテスト実行

### 6-2. ポーリングの最適化

**現状:** 審議完了後も3秒ポーリングが続く。
**改善:** 完了したらポーリング停止。実行中は2秒、完了後は停止、エラー時は指数バックオフ。

### 6-3. エラーバウンダリ

**現状:** コンポーネントのエラーでページ全体がクラッシュする可能性。
**改善:** React Error Boundary を各セクションに設置。エラー時はフォールバックUIを表示。

### 6-4. デプロイの自動化

**現状:** 手動で `gcloud run deploy` を実行。
**改善:** GitHub Actions で main push 時に自動デプロイ。Staging/Production の環境分離。
